## 模块一：Agent 核心架构与底层原理

### 1.1 Agent = 大脑 + 记忆 + 工具 + 护栏

```
┌─────────────────────────────────────────────────┐
│                   User Input                     │
├──────────┬──────────┬──────────┬─────────────────┤
│ Planning │  Memory  │  Tools   │   Guardrails    │
│ (大脑)   │  (记忆)  │ (工具)   │   (护栏)        │
├──────────┴──────────┴──────────┴─────────────────┤
│                  LLM Backbone                    │
└─────────────────────────────────────────────────┘
```

**一句话定义**：Agent 是"以 LLM 为决策核心，能自主规划、调用外部工具、管理上下文记忆、并受安全护栏约束"的智能执行体。

与传统 Chatbot 的本质区别：
| 维度 | Chatbot | Agent |
|------|---------|-------|
| 执行路径 | 预定义流程 / 意图槽位 | 动态规划，运行时决定下一步 |
| 工具调用 | 无或硬编码 | 通过 Function Calling 动态选择 |
| 记忆管理 | 仅 Context Window | 短期 + 长期记忆（RAG） |
| 失败恢复 | 返回兜底话术 | 可自主重试、降级、Self-Correction |

### 1.2 大脑（Planning）：CoT、ReAct、Task Decomposition

#### 1.2.1 CoT（Chain-of-Thought，思维链）

**原理**：通过在 Prompt 中加入"Let's think step by step"或提供推理示例，引导 LLM 将复杂问题拆解为中间推理步骤，而非直接输出结论。

**为什么有效**：LLM 本质是 next-token prediction。当要求它直接输出最终答案时，中间推理被压缩成隐式计算，容易出错。CoT 将隐式推理显式化，每一步的 token 生成都以前一步为条件，降低单步推理难度。

**PM 视角的应用**：
- 在入院文书 Agent 中，要求模型先输出"信息提取计划"（哪些字段需要从哪段录音中提取），再逐字段生成——这就是 CoT 的产品化落地。
- 好处：中间步骤可审计，出错时能精确定位是"提取错误"还是"生成错误"。

**面试话术**：
> "CoT 不是 Prompt 技巧，而是一种将 LLM 的隐式推理显式化的工程方法。它的产品价值在于——让 AI 的决策过程可解释、可审计、可调试。"

#### 1.2.2 ReAct（Reasoning + Acting）

```
循环：
  Thought → 我需要查询患者的过敏史
  Action  → 调用 Tool: query_allergy_history(patient_id="P-001")
  Observation → 返回：青霉素过敏
  Thought → 过敏史已获取，下一步生成主诉字段
  Action  → 调用 Tool: generate_chief_complaint(...)
  ...
```

**核心逻辑**：LLM 在每一步先"想"（Reasoning），再"做"（Acting），然后根据外部反馈（Observation）决定下一步。这是一个闭环：推理 → 行动 → 观察 → 再推理。

**与纯 CoT 的区别**：
- CoT 是"只想不做"——纯文本推理链，不与外部交互
- ReAct 是"边想边做"——每一步推理都可以触发工具调用，获取真实数据

**PM 视角**：
- ReAct 适合需要"动态决策"的场景：执行路径无法预定义、需要根据中间结果调整策略
- **我的入院文书 Agent 选的是 Plan-and-Execute 而非纯 ReAct**——因为入院文书的执行顺序有依赖（先取数据 → 再生成 → 再校验），Plan-and-Execute 先生成完整计划再逐步执行，每步失败可单独重试

#### 1.2.3 Task Decomposition（任务拆解）

三种常见拆解模式：

| 模式 | 描述 | 适用场景 | 我的项目映射 |
|------|------|----------|-------------|
| **Plan-and-Execute** | 先生成完整计划，再逐步执行 | 步骤有依赖、需要全局视野 | 入院文书 Agent（取数据→生成→校验） |
| **ReAct 动态决策** | 每步根据观察决定下一步 | 路径不确定、需要灵活应对 | 查房记录 Agent（根据当天检查结果决定关注点） |
| **DAG 预定义** | 固定有向无环图 | 流程标准化、确定性要求极高 | 医保编码校验（固定规则链） |

**面试话术**：
> "选哪种规划模式不是技术偏好，而是业务约束决定的。入院文书有明确的步骤依赖，所以我选 Plan-and-Execute；如果是开放式的问诊对话，ReAct 更合适；纯规则校验不需要 LLM 规划，用 DAG 就够了。"

### 1.3 记忆（Memory）：从 Context Window 到 RAG

#### 1.3.1 记忆分层架构

```
┌───────────────────────────────────────┐
│    Working Memory（工作记忆）          │  ← 当前对话上下文
│    = Context Window 内的内容           │  ← 生命周期：单次对话
├───────────────────────────────────────┤
│    Episodic Memory（情景记忆）         │  ← 近期交互摘要
│    = 对话摘要 / 会话历史              │  ← 生命周期：天/周级别
├───────────────────────────────────────┤
│    Semantic Memory（语义记忆）         │  ← 长期知识库
│    = Vector DB + RAG 按需检索         │  ← 生命周期：持久化
└───────────────────────────────────────┘
```

**我的项目映射**（科大讯飞入院文书）：
- **Working Memory**：当前对话的上下文（医患对话录音转写文本）
- **Episodic Memory**：当天诊疗周期的摘要（本次住院期间的检查结果汇总）
- **Semantic Memory**：患者长期病史，通过 RAG 按需检索

**关键设计细节**：
- 过敏史、手术史打 `immutable: true` 标签，不参与任何衰减或擦除
- Episodic Memory 每 24h 乘以衰减系数 0.85 进行软擦除
- 出院超过 90 天无访问，Episodic Memory 归档到冷存储

#### 1.3.2 RAG（Retrieval-Augmented Generation）全流程

```
用户查询
  │
  ▼
① Query 理解 ──→ 意图识别 + 查询改写（Query Rewriting）
  │
  ▼
② 检索（Retrieval）
  │  ├─ 稀疏检索：BM25（关键词匹配，适合精确术语）
  │  └─ 稠密检索：Embedding 相似度（语义匹配，适合自然语言描述）
  │  └─ 混合检索：Hybrid = BM25 + Dense（生产环境主流方案）
  │
  ▼
③ 重排序（Reranking）
  │  ├─ Cross-Encoder 模型对 Top-K 结果精排
  │  └─ 过滤相关度 < 阈值的 chunk（我设的阈值是 0.4）
  │
  ▼
④ 上下文组装（Context Assembly）
  │  ├─ 将检索到的 chunks 按相关度排序
  │  └─ 拼接进 Prompt（注意 Token 预算控制）
  │
  ▼
⑤ 生成（Generation）
  │  └─ LLM 基于原始 Query + 检索上下文生成回答
  │
  ▼
⑥ 后处理（Post-process）
     ├─ 幻觉检测：输出中的具体数值回查来源
     └─ 溯源标注：标记每条信息的来源 chunk
```

#### 1.3.3 RAG 常见问题与产品优化策略

**问题一：检索不准（Recall 低）**

| 优化策略 | 具体做法 | 我的项目实践 |
|----------|----------|-------------|
| **Query Rewriting** | 用 LLM 对用户 Query 做改写/扩展，增加检索召回 | 医生口语"血压一直高"→改写为"高血压病史+血压控制情况" |
| **Hybrid Search** | BM25 + Dense 混合检索，兼顾精确匹配和语义匹配 | 医学术语用 BM25 精确匹配，症状描述用 Dense 语义匹配 |
| **Chunk 策略优化** | 调整分块大小、重叠窗口；按文档结构分块而非固定长度 | 病历按"字段级别"分块（主诉一个 chunk、现病史一个 chunk），而非按 512 token 切 |

**问题二：检索到了但生成时忽略（Faithfulness 低）**

| 优化策略 | 具体做法 |
|----------|----------|
| **Prompt 工程** | 明确指令"仅基于以下上下文回答，如无相关信息请回答'信息不足'" |
| **Citation 机制** | 要求模型在输出中标注引用来源（如 [来源1]），便于验证 |
| **Reranking 前置** | 用 Cross-Encoder 过滤低质量 chunk，减少噪声干扰 |

**问题三：生成幻觉（Hallucination）**

| 优化策略 | 具体做法 | 我的项目实践 |
|----------|----------|-------------|
| **双 Agent 校验** | 主 Agent 生成，辅 Agent 做事实核查 | 入院文书：主 Agent 生成，辅 Agent 做合规审计 |
| **输出溯源** | 对每个具体数值回查来源 chunk | 药物剂量无法溯源 → 标记为"[待验证]" |
| **保守策略** | "未提及则输出未提及，不推理" | 我在讯飞坚持的核心规则——宁可字段空白，不可幻觉填充 |

**面试话术**：
> "RAG 不是接了向量数据库就完事了。检索不准要优化 Query Rewriting 和 Chunk 策略；检索到了但生成偏了要优化 Prompt 和 Reranking；最终生成的幻觉要靠双 Agent 校验和输出溯源兜底。这是三层不同的问题，对应三层不同的优化策略。"

### 1.4 工具（Tools）：Function Calling 的工作原理

#### 1.4.1 Function Calling 核心机制

```
用户输入: "帮我查一下张三的过敏史"
    │
    ▼
LLM 判断需要调用工具，输出结构化 JSON：
{
  "function": "query_allergy_history",
  "parameters": {
    "patient_id": "P-001",
    "patient_name": "张三"
  }
}
    │
    ▼
系统层解析 JSON → 调用实际 API → 获取结果
    │
    ▼
将结果注入 Context → LLM 继续生成最终回答
```

**本质**：Function Calling 不是 LLM "执行"了函数，而是 LLM 输出了一段结构化的"调用意图"，由系统层（Orchestrator）负责实际执行。LLM 只做决策，不做执行。

#### 1.4.2 工具原子化设计原则

**我的实践**：每个工具有独立的入参 Schema 和返回值 Schema。

```json
{
  "name": "generate_chief_complaint",
  "description": "根据医患对话提取主诉字段",
  "parameters": {
    "type": "object",
    "properties": {
      "transcript": { "type": "string", "description": "医患对话转写文本" },
      "max_symptoms": { "type": "integer", "default": 3 }
    },
    "required": ["transcript"]
  },
  "returns": {
    "chief_complaint": "string",
    "confidence": "float",
    "source_segments": "array"
  }
}
```

**原子化的好处**：
1. 每个工具独立可测试、独立可替换
2. 工具之间通过标准 Schema 通信，降低耦合
3. 便于构建工具白名单——角色权限绑定到具体工具

#### 1.4.3 容错机制（三层架构）——PredicTx 多租户实战

在 PredicTx 项目里，多租户架构下的工具调用容错是一个非常现实的工程挑战。我们接入了医疗专用模型和开源模型，不同模型生成 Function Calling 参数的质量差异很大（OpenMistral 在某些边缘 case 下会生成格式不合规的 JSON）。

**Layer 1：调用前参数校验（Pre-call Validation）**

在 Tool 被调用之前，有一个参数验证层，对照 Tool 的 JSON Schema 做严格校验。定义四类错误：
- `MISSING_REQUIRED`：缺少必填参数 → 触发 Self-Correction
- `TYPE_MISMATCH`：类型错误（如 patient_id 传了 string 而不是 int）→ 触发类型强转，强转失败则 Self-Correction
- `OUT_OF_RANGE`：值域越界 → 直接拒绝，返回错误描述
- `SCHEMA_INVALID`：JSON 格式本身损坏 → 触发 JSON Repair（用规则修复常见截断错误）

**Layer 2：重试与退避（Retry with Exponential Backoff）**

```
第1次失败 → 等待 1s 重试
第2次失败 → 等待 2s 重试
第3次失败 → 等待 4s 重试
第4次失败 → 触发降级策略
全局最大重试次数 = 3，超出后不再重试，防止死循环。
```

**Layer 3：降级与 Fallback**
- **软降级**：切换到备用 Tool（如主 EMR 接口不通，切备用只读 API 拿缓存数据）
- **硬降级**：所有 Tool 均失败，返回"当前系统无法完成此操作，请联系 XX"的标准化错误，并记录完整的失败链路到日志

#### 1.4.4 Self-Correction 机制

当参数校验失败时，不是直接报错，而是构建一个 Self-Correction Prompt，把错误信息和原始参数一起喂给模型：

```
你上一次调用工具的参数存在错误：
- 错误类型：MISSING_REQUIRED
- 缺少字段：patient_id
- 原始输入：{"doc_type": "admission", "doctor_id": "D-001"}
请根据用户的原始请求重新生成完整的参数。
原始请求：[用户输入]
```

实测 Self-Correction 在参数缺失类错误上的修复成功率达到约 **78%**，大幅减少了需要人工介入的频次。

#### 1.4.5 幂等性设计

医疗写操作全部要求幂等。每次写操作调用都携带 `idempotency_key`（由 `task_id + tool_name + timestamp` 生成），服务端去重，确保因重试导致的重复调用不会产生重复写入。

**面试话术**：
> "工具容错不是'加个 try-catch'那么简单。我设计了三层：调用前用 Schema 校验 + Self-Correction 自动修复参数错误（修复率 78%），调用中用指数退避重试，调用后用软硬两级降级兜底。同时所有写操作都要求幂等——重试不能产生重复写入。"

### 1.5 局限性管理（PM 必知）

#### 1.5.1 Context Window 限制

| 问题 | 产品影响 | 优化策略 |
|------|----------|----------|
| 输入超长被截断 | 关键信息丢失 | ① 摘要压缩：对历史对话做递进式摘要<br>② 分层记忆：只把当前相关内容放入 Context<br>③ RAG 按需检索：不把全量知识塞进 Prompt |
| 输出被截断 | 生成不完整 | ① 分段生成：将长文档拆成多次生成<br>② 结构化输出：先生成大纲，再逐段填充 |

#### 1.5.2 Token 成本控制

| 策略 | 描述 | 我的实践 |
|------|------|----------|
| **语义缓存** | 相似请求命中缓存，不重复调用 LLM | 高频标准化入院场景，相似度 > 阈值直接返回缓存 |
| **意图前置分流** | 用轻量分类模型过滤简单请求 | 低复杂度请求不进入完整 Agent 链路 |
| **Prompt 精简** | 去掉冗余指令，用 few-shot 替代长指令 | 系统 Prompt 从 2000 token 优化到 800 token |
| **模型分级** | 简单任务用小模型，复杂任务用大模型 | 意图分类用轻量模型，文书生成用大模型 |

#### 1.5.3 响应延迟优化

| 策略 | 效果 |
|------|------|
| **Streaming 输出** | 首字延迟（TTFT）从秒级降到百毫秒级 |
| **意图前置** | 过滤 60%+ 的简单请求，不走完整 Agent 链路 |
| **语义缓存** | 命中缓存的请求延迟接近 0 |
| **并行工具调用** | 无依赖的工具并行执行，减少串行等待 |

**面试话术**：
> "作为 PM，我关注三个核心约束：Context Window 决定了我能给模型多少信息，Token 成本决定了我的毛利模型，延迟决定了用户体验。这三者互相制约——给更多 Context 提升质量，但增加成本和延迟。产品设计的核心就是在这三者之间找到平衡点。"

## 模块二：平台选型与市场趋势

### 2.1 公共平台 vs 企业自研平台

| 维度 | 公共平台（Coze/Dify） | 企业自研平台 | PM 决策建议 |
|------|----------------------|-------------|-------------|
| **数据安全** | 数据经过第三方服务器；部分支持私有化部署但成本高 | 数据完全自主可控；满足等保/HIPAA等合规要求 | 医疗、金融等强合规行业必须自研或私有化部署 |
| **定制化深度** | 提供可视化编排 + 插件市场，覆盖 80% 通用场景；深度定制受限（如自定义记忆策略、多 Agent 通信协议） | 架构完全可控，可实现任意定制（如我设计的双 Agent 校验机制、DFA 状态机护栏） | 如果业务需要"工具白名单+角色权限绑定+自定义通信协议"级别的定制，公共平台做不到 |
| **开发成本** | 低——拖拽建 Agent，几小时出 Demo | 高——需要搭建编排层、记忆管理、工具注册、评测体系 | MVP / PoC 阶段用公共平台验证可行性；规模化后迁移自研 |
| **SaaS 多租户隔离** | 平台级隔离，但 Prompt/知识库的租户隔离粒度有限 | 可做到租户级 Prompt 隔离、知识库物理隔离、模型路由隔离 | B2B SaaS 产品（如 PredicTx）需要租户级隔离，公共平台难以满足 |

**面试话术**：
> "公共平台和自研不是二选一，而是阶段性选择。PoC 阶段用 Coze/Dify 快速验证——证明这个场景 Agent 能做；一旦验证通过要上生产，数据安全、多租户隔离、护栏定制这三个需求会迫使你走向自研。我在科大讯飞做的就是后者——医疗合规场景不可能把患者数据发到第三方平台。"

### 2.2 开源框架 vs 原生 API 开发

| 维度 | 开源框架（LangChain / AutoGPT） | 原生 API 开发 |
|------|-------------------------------|--------------|
| **上手速度** | 快——丰富的预建组件（Chain, Agent, Memory, Retriever） | 慢——需要从零搭建编排逻辑 |
| **灵活性** | 中——受框架抽象层约束，某些定制需 hack 底层 | 高——完全自由，不受框架限制 |
| **可维护性** | 框架更新频繁，升级可能 breaking change | 自己维护，版本可控 |
| **调试难度** | 抽象层多，错误堆栈深，调试困难 | 调用链清晰，问题定位直接 |
| **生产可用性** | LangChain 生产案例增多但仍有争议；AutoGPT 更偏实验性 | 大厂主流选择（OpenAI, 字节内部） |

**PM 决策框架**：
- **选框架的场景**：快速原型、技术验证、小规模内部工具
- **选原生 API 的场景**：生产环境、高确定性要求、需要深度定制编排逻辑
- **折中方案**：用框架做 PoC，生产时提取核心逻辑用原生 API 重写

### 2.3 Skills（原子能力）的封装复用

**定义**：Skill = 一个原子化的、可复用的能力单元，有明确的输入输出 Schema。

```
Agent = Planner + Memory + [Skill_1, Skill_2, ..., Skill_N]

例：入院文书 Agent 的 Skill 拆解：
  - Skill_1: 语音转写（ASR）
  - Skill_2: 实体抽取（NER）
  - Skill_3: 主诉生成
  - Skill_4: 现病史生成
  - Skill_5: 医保编码校验
  - Skill_6: 合规审计
```

**封装复用的产品价值**：
1. **跨 Agent 复用**：Skill_5（医保编码校验）同时被入院文书 Agent 和出院小结 Agent 使用
2. **独立迭代**：优化 Skill_2 的 NER 准确率，不影响其他 Skill
3. **权限管控**：不同角色可调用的 Skill 白名单不同

**我在讯飞的实践**：
> 在 Agent 工作空间设计中，我定义了三级权限和 Skill 复用机制，使 Agent 开发效率提升 15%、Skill 复用率提升 30%。核心设计是：所有者定义 Skill 库，管理员决定哪些 Skill 可被工作空间内的 Agent 调用，成员只能使用已授权的 Skill 组合搭建 Agent。

## 模块三：确定性与数据评测

### 3.1 核心评测指标体系

| 指标 | 定义 | 计算方式 | 行业基准 | 产品意义 |
|------|------|----------|----------|----------|
| **满意度（Satisfaction）** | 用户对 Agent 输出的主观满意程度 | Thumbs Up 数 / 总交互数 | >80% 为及格 | 最终的北极星指标，但滞后且主观 |
| **成功率（Task Success Rate）** | Agent 完成用户意图的比例 | 成功完成的任务数 / 总任务数 | 因场景差异大，通常 >70% | 衡量 Agent 是否"能用" |
| **采纳率（Adoption Rate）** | 用户实际采纳 Agent 输出的比例 | 被采纳的输出数 / 总输出数 | 我的项目达到 80% | 衡量 Agent 是否"好用"——比成功率更严格 |
| **幻觉率（Hallucination Rate）** | Agent 输出中无法溯源的信息比例 | 含幻觉的输出数 / 总输出数 | <5% 为良好 | 高风险场景（医疗/金融）的核心安全指标 |
| **Token 消耗率** | 每次交互的平均 Token 消耗 | 总 Token / 总交互次数 | 因场景差异大 | 直接影响成本和毛利率 |
| **首字延迟（TTFT）** | 从用户发送请求到收到第一个 token 的时间 | 端到端测量 | <2s 为良好 | 用户体感延迟的核心指标 |

**指标间的关系**：
```
Token 消耗 ↑ ──→ 成本 ↑ ──→ 毛利 ↓
     │
     └──→ 延迟 ↑ ──→ 满意度 ↓

幻觉率 ↓ ──→ 信任度 ↑ ──→ 采纳率 ↑ ──→ 满意度 ↑
```

### 3.2 RAGAS 框架详解

RAGAS（Retrieval Augmented Generation Assessment）是目前业界最主流的 RAG 系统评测框架。

| RAGAS 指标 | 评估对象 | 含义 | 低分时的产品行动 |
|-----------|---------|------|----------------|
| **Faithfulness（忠实度）** | 生成层 | 生成内容是否忠实于检索到的上下文？ | 优化 Prompt 约束 + 加强 Citation 机制 |
| **Answer Relevance（回答相关性）** | 生成层 | 生成内容是否回答了用户的问题？ | 优化 Query 理解 + Prompt 中强调"回答用户问题" |
| **Context Relevancy（上下文相关性）** | 检索层 | 检索到的上下文是否与问题相关？ | 优化检索策略 + Reranking + Chunk 策略 |
| **Context Recall（上下文召回）** | 检索层 | 回答问题所需的信息是否都被检索到了？ | 优化 Embedding 模型 + 扩展检索范围 + Query Rewriting |

**我的项目实践**：
- 在科大讯飞向量数据库迁移中，用 RAGAS 的 **Context Relevancy** 实时对比新旧两套检索结果
- 设定硬性切换条件：新库评分连续三天不低于旧库 + P99 延迟改善 → 才自动切流量
- 这套方案实现了业务零感知的平滑迁移

**面试话术**：
> "RAGAS 的价值不在于它给你一个分数，而在于它把'RAG 效果不好'这个模糊的问题拆成了四个可独立优化的维度。Context Recall 低说明检索漏了信息，Context Relevancy 低说明检索到了噪声，Faithfulness 低说明模型没忠实于检索结果，Answer Relevance 低说明模型答非所问。每个维度对应不同的优化动作。"

### 3.3 评测闭环：Golden Set + LLM-as-a-Judge + Badcase 回流

```
┌──────────────────────────────────────────────────┐
│              评测闭环完整流程                       │
│                                                    │
│  ① 构建 Golden Set（黄金测试集）                    │
│     ├─ 来源 1: 人工标注（专家定义正确输出）          │
│     ├─ 来源 2: 线上 Badcase 回流（用户点踩的累积）   │
│     └─ 来源 3: 边缘 Case 补充（对抗性测试用例）      │
│                                                    │
│  ② 离线评估                                        │
│     ├─ 自动化：RAGAS 跑全量 Golden Set              │
│     └─ LLM-as-a-Judge: 用 GPT-4 级模型给输出打分   │
│                                                    │
│  ③ 在线监控                                        │
│     ├─ 采纳率 / 修改率 / 重新生成率                  │
│     └─ 用户点踩 → 自动触发 Badcase 收集             │
│                                                    │
│  ④ Badcase 闭环                                    │
│     ├─ 自动打标签（分类 Badcase 类型）              │
│     ├─ 自动聚类（相似 Badcase 合并）                │
│     ├─ 优先级排序（影响用户数 × 频次）              │
│     ├─ 归因分析（检索层 / 生成层 / 意图层）          │
│     ├─ 修复（改 Prompt / 改知识库 / 改 Reranker）   │
│     ├─ A/B Test（新旧版本对照）                     │
│     └─ 灰度发布 → 全量 → Badcase 补充到 Golden Set  │
└──────────────────────────────────────────────────┘
```

**LLM-as-a-Judge 的使用方法**：
- 用强模型（如 GPT-4）作为"裁判"，对 Agent 输出按多个维度打分
- 优点：比人工标注快 100 倍，成本低
- 缺点：裁判模型本身有偏差（position bias, verbosity bias）
- **补偿方案**：定期用人工标注结果校准裁判模型的打分偏差

**面试话术**：
> "AI Agent 的评测不是上线前做一次就完了。Golden Set 提供离线基准，线上监控捕捉真实问题，Badcase 闭环确保每一个被用户标记的问题都有归因和修复。这三者构成一个持续迭代的飞轮——线上 Badcase 回流到 Golden Set，让下一次离线评估覆盖更多真实场景。"

## 模块四：简历深度对齐（模拟面试场景）

### 4.1 入院文书 Agent 工作流重构叙事

**面试时的完整叙事框架（5 步法）**：

**Step 1 — 场景判断**：
> "第一步判断这个场景是否真的需要 Agent。入院文书：医生输入完全开放、执行路径无法预定义、需要跨多个系统取数——三个条件都满足，才值得做 Agent。如果只是模板填空，规则系统就够了。"

**Step 2 — 架构选型**：
> "选 Plan-and-Execute 架构。因为入院文书有步骤依赖：先取数据 → 再生成 → 再合规校验。先生成完整执行计划，再逐步执行，每步失败可单独重试。"

**Step 3 — 工具链 + 记忆设计**：
> "每个工具原子化，有独立的入参 Schema 和返回值 Schema。记忆分三层：Working Memory 放当前对话、Episodic Memory 放当天摘要、Semantic Memory 通过 RAG 按需检索。关键写入和擦除逻辑要定义清楚——过敏史打 immutable 标签。"

**Step 4 — 护栏设计（三层）**：
> "Pre-process：权限校验 + PII 脱敏 + 意图安全检测。Runtime：DFA 状态机限制合法转移路径 + Step Limit = 15。Post-process：幻觉检测 + 合规扫描 + Human-in-the-loop 触发条件。核心原则：高风险写操作必须经过医生确认。"

**Step 5 — 评测体系**：
> "离线：Golden Set 评估。在线：监控采纳率和修改率。用户点踩触发 Badcase 自动聚类 → 归因分析 → 修复 → A/B Test → 灰度发布的完整闭环。"

### 4.2 多 Agent 协同冲突：如何回答

**面试官问**："你的双 Agent 校验机制，两个 Agent 意见冲突时怎么办？"

**标准回答**：
> "我设计了一套通信协议，消息格式包含 sender、task_id、payload 和 priority 字段。冲突时有明确的三级优先级：
> 1. **合规规则最高**：医保编码这类硬约束，合规 Agent 说不行就不行
> 2. **医生确认其次**：医生的主动修改覆盖 Agent 建议
> 3. **Agent 建议最低**：生成 Agent 的输出是建议，不是决策
>
> 一旦涉及合规硬约束的冲突，强制触发人工介入，Agent 不能自动改。这个设计的核心原则是——Agent 永远不是最终决策者。"

### 4.3 五个高难度面试题 + 标准回答

#### 陷阱题 1：「你说采纳率 80%，那剩下 20% 是什么情况？你分析过吗？」

**考察点**：数据敏感度——你是否只看正向指标，忽略了失败分析。

**标准回答**：
> "20% 未采纳分为三类：
> - **约 10%** 是低风险字段的格式偏差（如婚育史的表述格式不符合医院习惯），医生选择手动修改而非完全拒绝——这部分通过逐医院的模板微调可以持续提升；
> - **约 5%** 是录音质量问题导致的源数据不准（噪声大、方言重），这不是 Agent 层的问题，需要在 ASR 层优化；
> - **约 5%** 是真正的生成质量问题（幻觉或逻辑错误）。
>
> 如果让我重来，我会在'重新生成'事件里加一个轻量的原因选择——内容不准 / 格式不对 / 其他——两秒钟的操作，但能让每次迭代有明确的优化靶点。这是我在这个项目里最大的遗憾。"

#### 陷阱题 2：「RAG 检索出来的内容经常不相关，你怎么排查和解决？」

**考察点**：是否有系统性的排查框架，而不是单点优化。

**标准回答**：
> "检索不相关要分层排查：
> 1. **先看 Embedding 质量**：用几个典型 Query 手动检查 Top-10 结果，如果语义明显不匹配，说明 Embedding 模型不适合这个领域——医疗场景我们换过领域微调的 Embedding 模型，效果显著提升。
> 2. **再看 Chunk 策略**：如果 Top-10 有相关 chunk 但排名靠后，说明 chunk 粒度有问题。我们把病历从固定 512 token 分块改为按字段级别分块后，Context Relevancy 提升了 15%。
> 3. **最后看 Query 质量**：用户的原始 Query 往往不是好的检索 Query。加 Query Rewriting（用 LLM 改写用户问题为更适合检索的表述）可以进一步提升。
>
> 排查顺序是：Embedding → Chunk → Query → Reranking，从底层往上排。"

#### 陷阱题 3：「你的 Agent 上线后，如果某天幻觉率突然飙升，你怎么应急？」

**考察点**：线上应急能力 + 系统性思维。

**标准回答**：
> "分三步：止血 → 定位 → 修复。
> 1. **止血（5分钟内）**：启动降级策略。对幻觉率超过阈值的场景，Agent 输出自动加上'[待人工确认]'标签，或直接切回规则引擎兜底。绝不能让有幻觉的输出直接到达用户。
> 2. **定位（1小时内）**：看幻觉是集中在某类字段还是全面爆发。如果集中——可能是某个知识库更新引入了脏数据；如果全面——可能是模型 API 侧出了问题（如上游模型版本更新）。
> 3. **修复**：脏数据问题回滚知识库版本；模型问题切换备用模型或锁定模型版本。修复后在 Golden Set 上跑回归测试，确认幻觉率恢复正常才解除降级。
>
> 核心是——上线前就要定义好幻觉率的监控阈值和自动降级触发条件，而不是事后人工处理。"

#### 陷阱题 4：「Coze 这类平台已经很成熟了，企业为什么还要自研 Agent？」

**考察点**：平台选型的深度理解。

**标准回答**：
> "Coze 解决的是 80% 的通用场景，但企业自研解决的是那关键的 20%。具体来说有三个 Coze 做不到的事：
> 1. **数据合规**：医疗场景的患者数据不能出域，金融场景的交易数据不能经过第三方服务器。私有化部署成本高，且 Coze 的私有化方案不支持深度定制。
> 2. **护栏定制**：我需要 DFA 状态机控制 Agent 行为路径、需要自定义的多 Agent 通信协议、需要三级优先级的冲突仲裁——这些在 Coze 的可视化编排里做不到。
> 3. **多租户隔离**：B2B SaaS 场景下，每个租户的 Prompt、知识库、模型路由都需要物理隔离。Coze 的租户隔离粒度不够。
>
> 但我不否认 Coze 的价值——PoC 阶段用 Coze 验证场景可行性是最快的。关键是要知道什么时候该从平台迁移到自研。"

#### 陷阱题 5：「你觉得 AI Agent 和 RPA 有什么区别？为什么不直接用 RPA？」

**考察点**：对 Agent 本质的理解深度。

**标准回答**：
> "RPA 是确定性自动化——它按预定义的规则和流程执行，输入确定则输出确定，适合结构化、重复性高的任务（如每天从系统 A 复制数据到系统 B）。
>
> Agent 是不确定性自动化——它能处理开放式输入，动态决定执行路径，能调用工具获取信息后再做判断。
>
> 关键区别在**输入的开放程度**和**执行路径的确定性**：
> - 如果输入是结构化的、路径是固定的 → 用 RPA，成本更低、确定性更高
> - 如果输入是自然语言、路径需要动态决策 → 用 Agent
>
> 入院文书为什么不能用 RPA？因为医生的口语描述千变万化，你无法预定义所有的提取规则。但医保编码校验可以用 RPA——输入和规则都是固定的。
>
> 实际产品里，Agent 和 RPA 往往是组合使用的——Agent 处理模糊输入并做决策，RPA 负责确定性的执行动作。"

## 模块五：术语纠错与话术优化

### 5.1 十大易错术语

| # | 小白说法 | 问题 | 专业 PM 表达 |
|---|---------|------|-------------|
| 1 | "大模型调用 API" | 主客体反了。大模型不调 API，是系统调大模型的 API | "系统通过 LLM API 调用模型推理能力" |
| 2 | "Agent 执行了 Function" | Agent 不执行函数，只输出调用意图 | "LLM 输出了 Function Calling 的结构化参数，由 Orchestrator 层执行实际调用" |
| 3 | "RAG 就是接了个向量数据库" | 把整个检索增强系统简化为存储层 | "RAG 是'检索-重排-生成'的端到端流程，向量数据库只是其中检索层的存储组件" |
| 4 | "用 Prompt 调教模型" | "调教"暗示在修改模型权重，实际没有 | "通过 Prompt Engineering 优化模型的输出行为"（如果是修改权重叫 Fine-tuning） |
| 5 | "Agent 有自主意识" | 拟人化描述，面试官会质疑你的技术认知 | "Agent 基于 LLM 的推理能力进行动态决策"——它是工具，不是生命 |
| 6 | "Embedding 就是把文字变成向量" | 过于简化，没体现核心价值 | "Embedding 将文本映射到语义空间，使语义相近的文本在向量空间中距离更近，这是 RAG 检索的基础" |
| 7 | "模型幻觉是因为模型太笨" | 错误归因 | "幻觉是 LLM 的结构性特征——它本质是概率性生成，不是知识检索系统。需要通过 RAG、Grounding、事实核查等工程手段来缓解" |
| 8 | "Fine-tuning 和 Prompt Engineering 差不多" | 本质区别 | "Prompt Engineering 是调整输入，模型不变；Fine-tuning 是用领域数据调整模型权重。前者零成本即时生效，后者需要数据+算力+时间" |
| 9 | "Token 就是一个字" | 不准确 | "Token 是模型的最小处理单元。中文大约 1 个字 ≈ 1.5-2 个 token，英文大约 1 个单词 ≈ 1-1.5 个 token。Token 数直接决定成本和 Context 容量" |
| 10 | "多 Agent 就是多调几次模型" | 忽略了协同和通信的复杂性 | "Multi-Agent 的核心不是'多个模型'，而是'分工与协同'——需要定义通信协议、冲突仲裁机制、共享记忆策略。多模型调用只是执行层，架构设计才是难点" |

### 5.2 高频对比概念速查

| 概念对 | 区别 | 一句话记忆 |
|--------|------|-----------|
| **引用率 vs 采纳率** | 引用率 = "有没有看"；采纳率 = "看了之后用了多少" | 引用率是曝光，采纳率是转化 |
| **Recall vs Precision** | Recall = 该找的找全了没；Precision = 找到的里面有多少是对的 | Recall 是广度，Precision 是精度 |
| **Fine-tuning vs RLHF** | Fine-tuning 用标注数据调权重；RLHF 用人类偏好反馈做强化学习 | Fine-tuning 是"教知识"，RLHF 是"教品味" |
| **Zero-shot vs Few-shot** | Zero-shot 不给示例直接问；Few-shot 给几个示例再问 | Zero-shot 是裸考，Few-shot 是开卷 |
| **Latency vs Throughput** | Latency = 单次请求的响应时间；Throughput = 单位时间处理的请求数 | Latency 是用户体验，Throughput 是系统容量 |
| **Context Window vs Knowledge Base** | Context Window 是模型单次能看的范围；Knowledge Base 是持久化的外部知识 | Context Window 是"桌面大小"，Knowledge Base 是"书架" |

### 6.1 性能优化完整叙事（入院文书全链路耗时压缩）

**面试叙事（第三部分技术攻坚之性能优化）**：

> "为了压缩从意图识别到文书生成的全链路耗时，我做了两件事：
>
> **第一，意图识别前置**：用轻量分类模型先过滤掉低复杂度请求，不让它进入完整的 Agent 链路。比如简单的'查看患者档案'请求，直接走规则路由返回，不需要经过 LLM 推理。这一步过滤了约 60% 的简单请求。

```
用户点踩（Thumbs Down）
    |
    v
+-- 1. 自动打标签（分类 Badcase 类型）
|      - 检索层问题：知识库没有相关内容 / 检索到了但不相关
|      - 生成层问题：检索正确但生成偏差 / 幻觉
|      - 意图识别问题：误解了用户意图
|
+-- 2. 自动聚类（相似 Badcase 合并）
|      - 避免重复修复同一类问题
|
+-- 3. 优先级排序（影响用户数 x 频次）
|      - 影响 100 个用户的低频问题 vs 影响 10 个用户的高频问题
|
+-- 4. 专家标注（定义正确输出）
|      - 医学专家标注"这个输入应该产生什么输出"
|
+-- 5. 归因分析（定位根因层）
|      - 检索层 → 改 Chunk 策略 / 改 Embedding / 补知识库
|      - 生成层 → 改 Prompt / 改 Reranker / 调温度参数

|
+-- 7. A/B Test（新旧版本对照测试）
|
+-- 8. 灰度发布 -> 全量上线
|
+-- 新产生的 Badcase 补充到 Golden Set，下次迭代使用
```

**面试话术**：
> "这套闭环的核心不是某一个环节，而是它形成了飞轮效应——用户的每一次点踩都会让系统变得更好，而不是被忽略。自动打标 + 聚类省掉了人工分类的成本，归因分析确保修复动作打在正确的层上（检索层问题去改 Prompt 没用），A/B Test 确保修复不会引入新问题。这是我在 PredicTx 搭建的、从 Demo 走向 Production 的关键基础设施。"

### 1.1 "主导 10 个临床场景 AI Agent 架构设计与落地"

#### 简历原话
> 分析用户行为数据，拆解 6 大高频临床场景断点；独立设计多 Agent 协作架构——定义入院文书 Agent 等 10 个 Agent 的意图识别规则、工具调用链路与上下文记忆策略，通过编排层实现跨 Agent 的病历数据流转与多轮对话状态管理；同步重塑移动/PC 组件交互体系，解决双端体验割裂问题。上线后，移动端 MAU 提升至 40%，入院记录及首次病程采纳率达 80%。

---

#### 追问 1：「你说'分析用户行为数据拆解 6 大断点'——数据从哪来？在你之前这些数据存在吗？你做了什么让这些数据变得可分析？」

**标准回答**：

**Situation**：系统有基础的日志埋点，但没有被设计成能回答业务问题的指标。比如"手动展开患者档案"这个行为，日志记录了点击事件，但没有记录"距上次自动关闭的间隔时间"这个属性——只知道有人点了，不知道在什么上下文下点的。

**Task**：让"有数据"变成"数据能回答问题"。

**Action**：
- 重新定义每个埋点事件需要携带的上下文属性——比如给"手动展开"加上"间隔时间"，给"开始录音"加上"当前门诊阶段（初诊/复诊）"，给"结束录音"加上"录音过程中是否中断"
- 属性加进去之后，同一个事件才能告诉你它发生在什么业务场景下
- 基于重定义后的事件流，搭建了漏斗分析：进入接诊 → 启动录音 → 完成录音 → 查看 AI 生成 → 采纳/拒绝

**Result**：识别出 6 大断点，其中最关键的是——移动端和 PC 端的操作路径逻辑完全不同，医生拿 Pad 在病房走到一半不知道怎么继续。

**避坑话术**：
> 绝对不能说"我看了数据就发现了问题"——面试官会追问"数据本来就在那，为什么之前没人发现？"正确的叙事是：**你重新定义了数据的采集方式，让数据从"有记录"变成"能诊断"**。

---

#### 追问 2：「10 个 Agent 之间的编排逻辑是什么？是串行还是并行？有没有环路？状态怎么传递？」

**标准回答**：

**Situation**：10 个 Agent 覆盖入院记录、首次病程、查房记录、出院小结等不同文书类型，它们之间并非完全独立——比如入院记录 Agent 生成的主诉和初步诊断，会作为首次病程 Agent 的输入。

**Task**：设计跨 Agent 数据流转机制，既要保证数据一致性，又不能让单个 Agent 失败拖垮整个链路。

**Action**：
- **编排模式**：不是简单的串行或并行，而是 **DAG（有向无环图）**。入院记录是根节点，首次病程和查房记录是下游节点，出院小结依赖所有前序节点。无环——设计时用拓扑排序验证，避免循环依赖
- **状态传递**：通过中央 State Store（类似 Redux 模式）。每个 Agent 执行完毕后，将输出写入 State Store，下游 Agent 从 State Store 读取。消息格式标准化：`{sender, task_id, payload, priority, timestamp}`
- **容错**：任一节点失败不阻塞无依赖的并行节点。比如查房记录 Agent 失败不影响医嘱 Agent 的执行

**Result**：整套 DAG 编排在 10 个 Agent 之间实现了病历数据的结构化流转，每个节点独立可重试。

**避坑话术**：
> 面试官听到"10 个 Agent"首先会质疑"你真的需要 10 个吗？为什么不用 1 个大 Agent？"——标准回答是：**单 Agent 的 Context Window 装不下 10 种文书类型的全部规则和示例，拆成 10 个是 Token 预算约束下的工程选择，不是为了炫技**。每个 Agent 的 System Prompt 专注于一种文书类型，准确率远高于通用 Agent。

---

#### 追问 3：「MAU 提升 40% ——你怎么证明这是你的贡献，而不是功能本身上线带来的自然增长？」

**标准回答**：

> "我不打算说这 40% 全是我的功劳——因为确实无法做 A/B 对照。但我可以说的是：没有我做的那件事，这 40% 完不成。"

**关键论据**：
1. **时间线论据**：V0.5 上线的入院记录和首程记录，算法团队在 V0.8 之前就已经交付了功能模块，但 MAU 没有同步增长。原因是移动端操作路径和 PC 端逻辑完全不同，医生拿 Pad 不知道从哪里进入接诊流程
2. **我做的事**：三级组件统一，核心解决的是"功能已有但不会用"的问题——算法团队造了发动机，我做的是把发动机装进一辆医生知道怎么开的车
3. **间接证据**："重新生成率"在 V0.8 后下降了——说明不是功能变强了，而是医生找到了正确的使用路径，使用质量提升了

**避坑话术**：
> 绝不能说"MAU 就是我的功劳"——面试官会直接否定。正确姿势是**主动承认无法隔离归因，但用时间线和间接证据证明你的工作是必要条件**。

---

#### 追问 4：「采纳率 80%，你的评测集量级多大？这个数字是怎么算的？引用率和采纳率有什么区别？」

**标准回答**：

**指标定义精确区分**：

| 指标 | 定义 | 计算方式 |
|------|------|---------|
| **引用率** | 医生是否查看了 AI 生成内容 | 查看次数 / 总生成次数 |
| **采纳率** | 医生是否将 AI 生成内容实际写入病历 | 被采纳的字段数 / 总生成字段数 |
| **修正率** | 医生采纳后是否做了修改 | 被修改的字段数 / 被采纳的字段数 |

**三者关系**：引用率反映"有没有看"，采纳率反映"用了多少"，修正率反映"用得满不满意"。

简历里写的 80% 是**采纳率**——字段级别的统计，覆盖入院记录约 20+ 个字段。评测集不是离线 Golden Set 的数字，而是线上真实使用数据：上线后累计数千份入院记录的实际采纳情况。

**避坑话术**：
> 混用"引用率"和"采纳率"会被懂数据的面试官直接质疑。必须能精确说出三者区别。

#### 追问 5：「你说'意图识别规则'，是基于规则的还是基于模型的？如果是规则，扩展性怎么办？」

**标准回答**：

**Action**：采用 **规则 + 轻量分类模型的混合架构**：
- **第一层：规则路由**——对明确关键词的请求直接分发。比如包含"入院记录"的请求直接路由到入院文书 Agent，不过模型。覆盖约 60% 的标准请求
- **第二层：分类模型**——对模糊请求（如"帮我整理一下刚才的对话"）走轻量 BERT 分类模型，判断应该路由到哪个 Agent
- **第三层：LLM 兜底**——分类模型置信度 < 0.7 时，用 LLM 做意图澄清（"您是想生成入院记录还是查房记录？"）

**扩展性设计**：新增一个 Agent 时，只需要：① 在规则层加关键词映射；② 在分类模型的训练集中补充样本并微调；③ 在 LLM 兜底的 Prompt 中增加一个选项。三层都有独立的扩展路径。

**避坑话术**：
> 不要说"全靠规则"——面试官会说规则不 scale。也不要说"全靠模型"——面试官会问成本和延迟。正确答案永远是**混合架构：规则覆盖高频确定性场景，模型处理长尾模糊场景**。

### 1.2 "医疗 Agent 平台多产线工作空间设计"

#### 简历原话
> 梳理账号管理断点，定义所有者/管理员/成员三级权限边界，构建三类资产共享管控机制，打通跨团队 Agent 复用路径；同步推进知识库 Excel 支持、节点扩展等两轮迭代。上线后，Agent 开发效率提升 15%，复用率提升 30%，重复建设成本大幅压缩。

---

#### 追问 1：「为什么三级权限而不是 RBAC？你不觉得这是偷懒吗？如果用户规模扩大后权限不够怎么办？」

**标准回答**：

**Situation**：有人提出做完整 RBAC，每个操作单独配权限，理由是将来要服务更多团队。

**Task**：在"权限灵活性"和"用户认知成本"之间做决策。

**Action——我的判断基于三个变量**：
1. **回到需求来源**：需求方原话不是要"精细控制"，而是要"边界感——知道自己的东西不会被人改"。RBAC 解决的不是这个问题
2. **用户结构**：不只是内部技术同学，还有医院 IT 管理员。几十个权限开关大概率配错，运营成本比功能本身更高
3. **产品阶段**：V0.9 首次上线，先验证协作机制能不能解决问题，比过度设计重要得多

**Result**：定义为"最小可行权限模型"而非终态。评审通过后产线负责人第一反应是"这就是我们需要的"，没人问为什么不能更细。

**避坑话术**：
> 关键句："权限越细越安全是个幻觉，细到用户配不对才是真正的风险。" 这句话展示的是**产品判断力**而非技术能力。

---

#### 追问 2：「'三类资产共享管控机制'——具体是哪三类？管控粒度是什么？」

**标准回答**：

三类资产：
1. **Agent 资产**：已搭建完成的 Agent 实例。管控粒度：所有者可编辑/删除，管理员可查看/复制，成员可调用
2. **知识库资产**：每个工作空间的私有知识库。管控粒度：跨工作空间只读引用，不可写入/修改他人知识库
3. **Skill/工具资产**：原子化能力插件。管控粒度：平台级 Skill 对所有工作空间可见，工作空间级 Skill 仅空间内可见

**核心设计原则**：**资产可见性 ≠ 资产可编辑性**。跨团队复用走"只读引用"，不走"完整复制"——避免同一个 Skill 出现多个版本分叉。

---

#### 追问 3：「复用率提升 30% 这个数字是怎么算的？」

**标准回答**：

**计算公式**：`复用率 = 被 ≥2 个工作空间引用的 Agent/Skill 数量 / 总 Agent/Skill 数量`

**基线**：上线前复用率接近 0——因为没有工作空间机制，各产线在同一命名空间下各建各的，无法发现和引用他人的 Agent。

**上线后**：跨工作空间的只读引用机制上线后，30% 的 Skill（主要是 NER 提取、医保编码校验等通用能力）被多个工作空间复用。

### 1.3 "MedEvidence 检索模块立项与智医助理集成"

#### 简历原话
> 横向对标 5 款国内外竞品，识别院内病历融合为差异化切入点；构建 0 重大错误率、引用有效率>95%、指南优先率>90% 的质量红线体系，确立 AI 功能准入标准。

---

#### 追问 1：「0 重大错误率——这在 AI 产品里现实吗？你怎么定义'重大错误'？」

**标准回答**：

**重大错误的精确定义**：输出中包含可能直接导致错误诊疗决策的信息。具体包括：
- 药物剂量错误（如将"5mg"写成"50mg"）
- 诊断方向性错误（如将恶性肿瘤误判为良性）
- 禁忌症遗漏（如患者有青霉素过敏但推荐了青霉素类药物）

**不是"0 错误率"，而是"0 重大错误率"**——低风险错误（如格式偏差、措辞不够规范）不计入此指标。

**实现手段**：
1. 输出层的幻觉检测——对所有具体数值（药物剂量、检查指标）回查来源，无法溯源的标记为"[待验证]"
2. 合规 Agent 扫描——针对禁忌症和药物相互作用做专项校验
3. Human-in-the-loop——涉及用药方案和手术建议强制推送人工审核

**避坑话术**：
> 面试官的本意是质疑"你是不是在吹牛"。关键是**精确定义什么叫'重大错误'**——把范围限缩到"可能直接导致诊疗事故"的错误，然后说明你用三层机制兜底。不要说"AI 永远不会出错"。

---

#### 追问 2：「引用有效率 >95%——如果检索到的文献是 5 年前的过时指南呢？你怎么保证引用的时效性？」

**标准回答**：

**Action**：
1. **知识库时效性管理**：指南类文档带入库时间戳和版本号，检索时对"最新指南"类查询自动加时间权重——优先返回最近 2 年的内容
2. **指南优先率 >90%**：这个指标就是用来解决这个问题的——要求 AI 输出中引用的来源，≥90% 必须是当前有效的临床指南（而非过期指南或非权威来源）
3. **过期内容标记**：被新版替代的旧版指南不从知识库删除（因为某些历史病例分析仍需要引用），但在检索结果中标记"[已有更新版本]"

**核心差异**：
| 差异维度 | 具体问题 |
|---------|---------|
| **API 格式** | OpenAI 兼容格式 vs 各厂商私有格式（如讯飞星火、百度文心的请求结构不同） |

```
应用层（Agent）

    ├─ OpenAI Adapter
    ├─ LLaMA Adapter
    ├─ Mistral Adapter
    └─ 讯飞星火 Adapter
    ↓
各模型原生 API
```

每个 Adapter 实现统一的 `ModelInterface`：`complete(prompt, params) -> Response`。应用层不需要知道底层是哪个模型。

## 四、技术短板"饱和式"补齐

---

### 4.1 RAG 工业级标准链路（深度版）

```
原始文档
  │
  ▼
① Parsing（文档解析）
  ├─ PDF/Word/HTML → 纯文本提取
  ├─ 表格识别（医疗场景大量表格数据）
  ├─ 图片 OCR（检查报告单的图片内容）
  └─ 难点：非结构化文档的结构还原
  │
  ▼
② Chunking（分块策略）
  ├─ 固定长度分块（512/1024 token）——简单但粗暴
  ├─ 语义分块（按段落/章节自然分割）——保持语义完整性
  ├─ 递归分块（先大块再小块，保留层级关系）
  ├─ 重叠窗口（相邻 chunk 有 10-20% 重叠）——防止信息断裂
  └─ 我的实践：病历按字段级别分块，每个字段一个 chunk
  │
  ▼
③ Embedding（向量化）
  ├─ 通用模型：OpenAI text-embedding-3-large / BGE-M3
  ├─ 领域微调：在医疗语料上微调 Embedding 模型
  ├─ 多语言：中英文混合场景需要多语言 Embedding
  └─ 维度选择：768/1024/1536 维——维度越高精度越高但成本越高
  │
  ▼
④ Indexing（索引构建）
  ├─ 向量索引：HNSW（快速近似最近邻搜索）
  ├─ 稀疏索引：BM25 倒排索引（关键词精确匹配）
  ├─ 混合索引：向量 + 稀疏 双路索引（生产环境主流）
  └─ 元数据索引：按文档类型/时间/来源等结构化过滤
  │
  ▼
⑤ Retrieval（检索）
  ├─ Dense Retrieval：Embedding 相似度搜索（语义匹配）
  ├─ Sparse Retrieval：BM25（关键词匹配）
  ├─ Hybrid Retrieval：两路融合（RRF / 加权融合）
  └─ 我的实践：医学术语走 BM25，症状描述走 Dense
  │
  ▼
⑥ Reranking（重排序）
  ├─ Cross-Encoder 精排（准但慢，只处理 Top-K）
  ├─ 相关度阈值过滤（<0.4 不进入 Context）
  └─ 多样性控制（避免 Top-K 结果过于相似）
  │
  ▼
⑦ Context Assembly（上下文组装）
  ├─ Token 预算分配（System Prompt + 检索上下文 + 用户 Query）
  ├─ 按相关度排序注入 Prompt
  └─ Lost-in-the-Middle 问题：关键信息放开头和结尾
  │
  ▼
⑧ Generation + Post-process（生成 + 后处理）
  ├─ LLM 基于上下文生成回答
  ├─ 幻觉检测：输出中具体数值回查来源
  ├─ Citation 标注：标记每条信息的来源 chunk
  └─ 合规过滤：禁用内容替换 + Human-in-the-loop
```

**在我简历项目中最容易出问题的环节**：

| 项目 | 最易出问题环节 | 原因 | 我的应对 |
|------|-------------|------|---------|
| 入院文书 Agent | ② Chunking | 医患对话是连续语音流，按固定长度切会打断关键信息 | 改为按字段级别分块 |
| MedEvidence | ⑤ Retrieval | 医学术语的同义词问题（如"高血压"vs"血压升高"vs"HBP"） | Hybrid 检索 + Query Rewriting |
| PredicTx 多租户 | ③ Embedding | 不同租户的医疗术语体系不同（中美医疗标准差异） | 租户级 Embedding 微调 |
| 教师策略推荐 | 不走 RAG | 知识库 <100 条，结构化程度高 | 直接结构化查询 |

### 4.2 长上下文与记忆管理（对比三种方案）

| 方案 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| **滑动窗口** | 只保留最近 N 轮对话，丢弃更早内容 | 实现简单、延迟低 | 丢失历史信息，多轮对话中可能"忘事" | 短对话、客服场景 |
| **递归摘要** | 超出窗口时，用 LLM 对历史内容做摘要压缩 | 保留关键信息、窗口利用率高 | 摘要本身可能丢失细节；每次摘要消耗额外 Token | 长对话、会议记录 |
| **向量数据库长期记忆（RAG）** | 历史信息持久化到 Vector DB，按需检索注入 | 理论上无限记忆、按需精准检索 | 检索延迟、检索准确率不保证 | 跨会话记忆、知识库 |

**我的项目中的选择**：

- **Working Memory**（当前对话）：滑动窗口 + 关键信息置顶（如患者姓名、科室始终保持在 Context 中）
- **Episodic Memory**（诊疗周期）：递归摘要——每天对当天诊疗记录做一次摘要，压缩进 Context
- **Semantic Memory**（长期病史）：向量数据库 + RAG 按需检索

**三者组合使用，不是互斥选择**。面试时需要强调这一点。

---

### 4.3 意图识别与多工具冲突处理（深度补充）

#### 多工具冲突场景

当 LLM 在一次推理中同时想调用多个工具时：

| 冲突类型 | 示例 | 处理策略 |
|---------|------|---------|
| **资源冲突** | 两个工具同时写同一个字段 | 加锁 + 优先级仲裁（合规 > 医生 > Agent） |
| **逻辑冲突** | 工具 A 的输出与工具 B 的输入矛盾 | DAG 依赖检查，强制串行执行有依赖的工具 |
| **并发冲突** | 同时调用外部 API 超过限流 | 并发控制 + 队列化 + 指数退避 |
| **参数冲突** | LLM 给同一工具生成了两组不同参数 | 取置信度更高的一组；若无法判断，触发 Self-Correction |

#### 参数解析失败的完整容错链

```
LLM 输出工具调用参数
  │
  ├─ JSON 格式正确？
  │    ├─ 是 → Schema 校验
  │    └─ 否 → JSON Repair（修复截断/多余逗号等常见错误）
  │              ├─ 修复成功 → Schema 校验
  │              └─ 修复失败 → Self-Correction Prompt
  │
  ├─ Schema 校验通过？
  │    ├─ 是 → 执行工具调用
  │    └─ 否 → 分类错误类型
  │              ├─ MISSING_REQUIRED → Self-Correction
  │              ├─ TYPE_MISMATCH → 强制类型转换 → 失败则 Self-Correction
  │              ├─ OUT_OF_RANGE → 直接拒绝
  │              └─ SCHEMA_INVALID → Self-Correction
  │
  └─ Self-Correction 成功？
       ├─ 是 → 重新执行
       └─ 否 → 降级策略（软降级 → 硬降级）
```

## 五、平台对比与产品决策逻辑（升级版）

### 5.1 深度对比矩阵

| 维度 | Coze | Dify | 企业自研 |
|------|------|------|---------|
| **权限隔离** | 平台级租户隔离；Prompt/知识库隔离粒度有限 | 支持 Workspace 级别隔离，优于 Coze | 完全自定义：租户级 Prompt/知识库/模型路由/工具白名单物理隔离 |
| **数据边界** | 数据经过字节服务器；国内有等保认证但海外医疗场景需确认 HIPAA | 支持私有化部署；社区版开源可自建 | 数据完全自主可控；可满足 HIPAA/等保三级等最严格合规 |
| **插件扩展性** | 丰富的插件市场 + 自定义插件；但插件间依赖管理弱 | 插件生态较 Coze 小但开源可扩展；支持自定义 Tool | 完全自定义工具注册机制；可做工具间依赖图管理和冲突检测 |
| **Token 成本控制** | 平台定价，有一定折扣但不透明；无法做租户级 Token 预算 | 私有化部署后模型选择自由度高；可控制 Token 消耗 | 完全自主：模型分级路由 + 语义缓存 + 意图前置分流，Token 成本可精细管控到租户/场景粒度 |
| **适用阶段** | PoC / 轻量级内部工具 / 个人项目 | PoC / 中小团队生产环境 / 对开源友好的企业 | 规模化生产 / 强合规行业 / 多租户 B2B SaaS |

### 5.2 "为什么不直接用 LangChain"——标准回答

**面试官问**：你做的很多事情 LangChain 都有现成组件，为什么要自己造轮子？

**标准回答**：

> "LangChain 在三个场景下不够用：

> **第一，调试成本**。LangChain 的抽象层太深——Chain 套 Chain 套 Agent，出了问题你要翻好几层源码才能定位。我们的医疗场景要求每一步可审计、可溯源，LangChain 的黑盒性和我们的透明性要求矛盾。

> **第二，性能控制**。LangChain 默认的 Memory 实现在长对话场景下 Token 消耗膨胀很快，我们需要自定义的三层记忆策略（Working + Episodic + Semantic），LangChain 的 Memory 抽象不支持这种灵活度。

> **第三，生产稳定性**。LangChain 更新非常频繁，API 经常 breaking change。在医院 24/7 运行的系统里，我们不能承受上游依赖突然 break 的风险。

> 但我不否认 LangChain 的价值——我们 PoC 阶段就是用 LangChain 快速验证可行性的，验证通过后才提取核心逻辑用原生 API 重写。"

## 六、RAGAS 评测专题（工业级深度）

### 6.1 四大指标的工程化落地

| RAGAS 指标 | 工程化计算方式 | 自动化部署 | 告警阈值（我的项目） |
|-----------|--------------|-----------|-------------------|
| **Faithfulness** | 将生成内容拆成 N 条 claim，逐条检查是否能从上下文中找到支持 | 每次生成后异步计算，结果写入评测数据库 | < 0.8 触发告警 |
| **Answer Relevance** | 用 LLM 基于回答反向生成 N 个问题，计算与原始问题的相似度均值 | 每小时聚合计算，看板展示趋势 | < 0.75 触发告警 |
| **Context Relevancy** | 计算检索上下文中与问题相关的句子占比 | 实时计算，新旧系统对比 | 新系统 < 旧系统连续 1 天触发回滚 |
| **Context Recall** | 将 Ground Truth 拆成 claim，检查每条是否能在上下文中找到 | 离线评估，每周跑 Golden Set | < 0.7 触发检索策略优化 |

### 6.2 用户反馈 → 测试集样本的自动化回流

```
用户反馈（点踩/点赞/修改）
  │
  ├─ 点踩 → 自动提取 {query, retrieved_context, generated_answer, user_feedback}
  │          → 自动归类（检索层/生成层/意图层）
  │          → 聚类后选代表性样本
  │          → 人工标注正确答案（或 LLM-as-a-Judge 自动标注）
  │          → 写入 Golden Set
  │
  ├─ 点赞 → 作为正样本补充 Golden Set 的正例覆盖度
  │
  └─ 修改 → 提取修改 diff，分析哪些字段被修改
            → 修改频率高的字段 → 定向优化该字段的生成规则
```

**关键设计**：Golden Set 不是静态的——每周自动从线上 Badcase 中补充新样本，每月人工 Review 剔除过时样本。这确保评测集始终反映真实的用户问题分布。

## 七、面试避坑与 RBP 专项准备

### 7.1 RBP 双重含义

| 含义 | 全称 | 面试中的应用 |
|------|------|-------------|
| **HR 层面** | Role-Based Permission / 价值观面 | 阿里系/字节系面试的 HR 轮，考察文化 fit、团队协作、价值观 |
| **技术层面** | Role-Based Prompting | 在 System Prompt 中为 Agent 设定角色，约束其行为边界 |

### 7.2 HR/RBP 面的标准应对框架

**常见问题 1**："你遇到过和团队意见不一致的情况吗？怎么处理的？"

**标准回答（用三级权限的故事）**：
> "在工作空间设计中，有人提出做完整 RBAC，我坚持三级权限。我没有直接反对，而是做了一件事——回到需求来源看用户原话。用户要的不是精细控制，是边界感。我用用户原话说服了团队，这不是我赢了，是用户赢了。"

**常见问题 2**："你最大的失败是什么？"

**标准回答（用"自动展开"的故事）**：
> "我在定义病历生成的展示逻辑时，做了一个错误判断——让病历自动展开。这是 C 端'减少点击=更好体验'的思维惯性。上线后数据显示手动关闭次数反而上升了。原因是医生在问诊时屏幕突然弹出内容，打断了注意力。正确的设计是角标提示、由医生主动查看。这让我理解了——减少点击不等于降低认知负荷。"

### 7.3 Role-Based Prompting 技术层面

**定义**：在 System Prompt 中为 Agent 赋予明确角色和行为边界。

**我的实践示例**（入院文书 Agent 的 System Prompt 结构）：
```
角色：你是一个专业的入院记录生成助手。
边界：
  - 你只能生成入院记录相关字段
  - 若信息不足，输出"未提及"，不可推理补充
  - 药物剂量必须有原始录音依据，否则标记"[待确认]"
输出格式：
  - 按字段输出：主诉、现病史、既往史、过敏史...
  - 每个字段附 confidence 分数和来源片段
```

**面试话术**：
> "Role-Based Prompting 不只是'告诉 AI 你是谁'，而是在 System Prompt 层面定义行为的硬约束。在医疗场景，这些约束和 DFA 状态机、工具白名单一起构成了三层护栏——Prompt 层约束意图，状态机约束路径，白名单约束能力。"
